分析

回傳的資料為object，而其中的
results->coupon->object_info->image->url
應該為兌換項目的圖片


推測＆發現

確定獎項為一張張的圖片

經由多數次的不同帳號測試(如上截圖)，推測應該是流水號

照片應該是流水號，而規則應該是「 P_GXXXXX.jpg 」

其中的XXXXX為數字，從00000 ~ 99999 都有可能

每次用戶端第一次載入這些圖片都會花些時間等待，而第二次幾乎不用等待，表示可能兌換項目有用快取機制

讚喔訪問這些圖片連token都不用

嘗試過程

ver 1

利用圖片是流水好的特點，跑個迴圈，逐一訪問每個有可能出現的圖片
大量的迴圈，網頁崩潰機率高、甚至會有死機的情況

=> 訪問真的不需要token就可以訪問到這些核心內容

=> 回傳的http state code 要不200 要不 404，沒有出現例如：401, 403, 429之類的
     這表示該API沒有權限驗證也沒有訪問次數限制，表示我們可以無限制的刷洗他們的API來得到主機中的資料。
     目錄爆破將可成為下一步的手法

A1: 使用資料庫作為快取，若嘗試到的連結有返回資料，那麼就將這筆資料的連結寫進資料庫，下次就可以不必花費很多時間再重新抓取曾經抓過的東西
A2: 使用Server端語言（如php）抓取資料，將處理大型迴圈的程式交給Server端處理，以減輕用戶端的負擔。後端抓取到資料後再以API的形式傳給前端顯示、處理

檢討

漏洞與建議

問題：API重要資料存取時沒有驗證用戶權限
建議解決方案：訪問該API時，應將token加上，以確保用戶權限並記錄存取時間

問題：API 無訪問次數限制 建議解決方案：以MVC架構的Laravel 建構之 API 來說， 可以使用中介層(Middleware)限制該用戶在存取資料的次數， 若發生不當存取行為可以對請求回傳 error 429
現在主流的API如facebook, instagram皆有防止爬蟲大量請求資源的機制

問題：App 快取機制bug
建議解決方案：此bug產生主要有3個原因
1. App端的重要資料未加密
2. 讀取的資料並沒有事先經過驗證，導致被替換也能夠正常使用
3. 要兌換餐點時，雖然Server端會記錄我換的東西但POS機端無法得知內容，僅能依據手機中的圖片出餐

問題：重要檔案（兌換圖資）的檔名易於猜測，加上API沒有訪問次數限制，黑客將可能使用目錄爆破的方式竊取主機中的檔案（如尚未上架的商品資料）
 建議解決方案：API連結應該採用動態產生方式，例如：
https://{api-url}/{user-token(hash)}/{event_id}
存取後連結即失效，若在一段時間沒有請求該資源，連結也應作廢
